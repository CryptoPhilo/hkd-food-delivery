# 한경 음식배달 서비스 - 사용자 업무 흐름 테스트 시나리오

## 1. 시나리오 개요

### 1.1 테스트 대상
- **역할**: 고객 (배달 주문자)
- **목적**: 음식 주문부터 배달 완료까지 전체 플로우 검증

### 1.2 사전 조건
- 시스템이 정상 운영 중
- 테스트용 식당 및 메뉴 데이터 존재
- SMS 발송 서비스 정상 작동

---

## 2. 시나리오 1: 첫 주문 (신규 사용자)

### 2.1 시나리오 설명
신규 사용자가 처음으로 음식을 주문하는 전체 과정

### 2.2 테스트 단계

#### Step 1: 전화 수신 및 SMS 발송
| 항목 | 내용 |
|------|------|
| **사전조건** | 사용자가 서비스 번호로 전화 |
| **동작** | 시스템이 발신번호 확인 후 SMS 발송 |
| **API** | `POST /api/v1/webhooks/sms` |
| **기대결과** | 사용자에게 주문 페이지 URL이 포함된 SMS 발송 |
| **검증항목** | SMS 발송 성공, URL 유효성 |

#### Step 2: 주문 페이지 접속
| 항목 | 내용 |
|------|------|
| **사전조건** | SMS로 받은 URL 클릭 |
| **동작** | 홈 페이지(식당 목록) 표시 |
| **페이지** | `/` (Home) |
| **기대결과** | 배달 가능한 식당 목록 표시, 영업시간 확인 |
| **검증항목** | 식당 목록 로드, 거리 표시, 영업 상태 |

#### Step 3: 식당 선택 및 메뉴 탐색
| 항목 | 내용 |
|------|------|
| **사전조건** | 홈 페이지에서 식당 목록 확인 |
| **동작** | 식당 클릭 → 메뉴 페이지 이동 |
| **페이지** | `/restaurant/[id]` |
| **기대결과** | 식당 상세 정보, 메뉴 목록, 장바구니 표시 |
| **검증항목** | 메뉴 가격, 품절 여부, 이미지 로드 |

#### Step 4: 장바구니 담기
| 항목 | 내용 |
|------|------|
| **사전조건** | 메뉴 페이지에서 메뉴 확인 |
| **동작** | 메뉴 추가 버튼 클릭, 수량 조절 |
| **페이지** | `/restaurant/[id]` |
| **기대결과** | 장바구니에 메뉴 추가, 소계 계산 |
| **검증항목** | 수량 변경, 가격 합계 정확성 |

#### Step 5: 결제 페이지 이동
| 항목 | 내용 |
|------|------|
| **사전조건** | 장바구니에 1개 이상 메뉴 |
| **동작** | "주문하기" 버튼 클릭 |
| **페이지** | `/checkout` |
| **기대결과** | 주문 요약, 배달 정보 입력 폼 표시 |
| **검증항목** | 주문 내역 정확성, 배달비 계산 |

#### Step 6: 배달 정보 입력
| 항목 | 내용 |
|------|------|
| **사전조건** | 결제 페이지 진입 |
| **동작** | 배달 주소, 연락처, 요청사항 입력 |
| **페이지** | `/checkout` |
| **기대결과** | 입력 정보 저장, 배달비 재계산 |
| **검증항목** | 주소 좌표 변환, 배달 가능 여부 |

#### Step 7: 결제 진행
| 항목 | 내용 |
|------|------|
| **사전조건** | 배달 정보 입력 완료 |
| **동작** | "결제하기" 버튼 클릭 |
| **API** | `POST /api/v1/orders` |
| **기대결과** | PortOne 결제 창 팝업, 주문 생성 |
| **검증항목** | 결제 금액 정확성, 주문 번호 생성 |

#### Step 8: 결제 완료
| 항목 | 내용 |
|------|------|
| **사전조건** | 카드 결제 성공 |
| **동작** | PortOne webhook 수신, 주문 상태 업데이트 |
| **API** | `POST /api/v1/webhooks/payment` |
| **기대결과** | 주문 상태 `pending`으로 변경, SMS 발송 |
| **검증항목** | 결제 정보 저장, 알림 발송 |

#### Step 9: 주문 확정
| 항목 | 내용 |
|------|------|
| **사전조건** | 결제 완료, 확정 링크 수신 |
| **동작** | SMS 링크 클릭 → 주문 확정 |
| **API** | `PUT /api/v1/orders/:id/confirm` |
| **기대결과** | 주문 상태 `confirmed`로 변경 |
| **검증항목** | 토큰 유효성, 상태 변경 |

#### Step 10: 배달 상태 확인
| 항목 | 내용 |
|------|------|
| **사전조건** | 주문 확정 완료 |
| **동작** | "내 주문 조회" 페이지에서 상태 확인 |
| **페이지** | `/my-orders` |
| **기대결과** | 주문 상태 타임라인 표시 |
| **검증항목** | 상태별 단계 표시, 예상 시간 |

#### Step 11: 배달 완료
| 항목 | 내용 |
|------|------|
| **사전조건** | 배달원이 배달 완료 처리 |
| **동작** | 주문 상태 `completed` 확인 |
| **페이지** | `/my-orders` |
| **기대결과** | 배달 완료 표시, 완료 알림 수신 |
| **검증항목** | 최종 상태, 완료 시간 |

## 3. 시나리오 2: 주문 취소

### 3.1 시나리오 설명
결제 후 주문 확정 전에 주문을 취소하는 경우

### 3.2 테스트 단계

#### Step 1: 주문 생성 (시나리오 1 Step 1~8 반복)

#### Step 2: 주문 취소 요청
| 항목 | 내용 |
|------|------|
| **사전조건** | 주문 상태 `pending` |
| **동작** | "주문 취소" 버튼 클릭 |
| **API** | `PUT /api/v1/orders/:id/cancel` |
| **기대결과** | 주문 상태 `cancelled`로 변경 |
| **검증항목** | 취소 사유 저장 |

#### Step 3: 환불 처리
| 항목 | 내용 |
|------|------|
| **사전조건** | 주문 취소 완료 |
| **동작** | PortOne 환불 API 호출 |
| **API** | `POST /api/v1/payments/cancel` |
| **기대결과** | 결제 취소, 환불 처리 |
| **검증항목** | 환불 금액, 환불 상태 |

#### Step 4: 취소 확인
| 항목 | 내용 |
|------|------|
| **사전조건** | 환불 처리 완료 |
| **동작** | 내 주문 조회 |
| **페이지** | `/my-orders` |
| **기대결과** | 취소된 주문 표시, 환불 완료 표시 |
| **검증항목** | 취소 사유, 환불 일시 |
