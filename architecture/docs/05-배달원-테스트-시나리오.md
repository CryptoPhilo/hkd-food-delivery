# 한경 음식배달 서비스 - 배달원 업무 흐름 테스트 시나리오

## 1. 시나리오 개요

### 1.1 테스트 대상
- **역할**: 배달원 (Driver)
- **목적**: 배달 접수부터 완료까지 전체 플로우 검증

### 1.2 사전 조건
- 배달원 계정 등록 완료
- 모바일 기기에서 페이지 접속 가능

---

## 2. 시나리오 1: 일일 업무 시작

### 2.1 시나리오 설명
배달원이 출근하여 업무를 시작하는 과정

### 2.2 테스트 단계

#### Step 1: 배달원 페이지 접속
| 항목 | 내용 |
|------|------|
| **동작** | 배달원 페이지 URL 접속 |
| **페이지** | `/driver` |
| **기대결과** | 전화번호 입력 화면 표시 |

#### Step 2: 본인 확인
| 항목 | 내용 |
|------|------|
| **동작** | 전화번호 입력 |
| **API** | `GET /api/v1/drivers/status/:phone` |
| **기대결과** | 배달원 정보 표시 (이름, 카드번호) |
| **검증항목** | 등록된 배달원 여부 |

#### Step 3: 업무 시작
| 항목 | 내용 |
|------|------|
| **동작** | "업무 개시" 버튼 클릭 |
| **API** | `POST /api/v1/drivers/start-duty` |
| **기대결과** | 상태 변경 (업무 중), 시작 시간 기록 |
| **검증항목** | isOnDuty = true |

#### Step 4: 대기 주문 확인
| 항목 | 내용 |
|------|------|
| **동작** | 대기 중인 주문 목록 조회 |
| **API** | `GET /api/v1/drivers/pending-orders` |
| **기대결과** | 배정 가능한 주문 목록 표시 |
| **확인항목** | 주소, 배달비, 요청사항 |

---

## 3. 시나리오 2: 주문 배정 및 수락

### 3.1 시나리오 설명
대기 중인 주문을 확인하고 배달을 수락하는 과정

### 3.2 테스트 단계

#### Step 1: 대기 주문 탐색
| 항목 | 내용 |
|------|------|
| **동작** | 주소/배달비 기준 주문 탐색 |
| **확인항목** | 배달 거리, 예상 소요 시간 |
| **판단기준** | 수익성, 접근성 |

#### Step 2: 주문 상세 확인
| 항목 | 내용 |
|------|------|
| **동작** | 주문 카드 클릭하여 상세 확인 |
| **확인항목** | 식당 위치, 배달 주소, 메뉴 구성 |
| **확인항목** | 고객 요청사항 |
| **확인항목** | 총 배달비 |

#### Step 3: 개별 주문 배정
| 항목 | 내용 |
|------|------|
| **동작** | "배정" 버튼 클릭 |
| **API** | `POST /api/v1/drivers/assign/:orderId` |
| **기대결과** | 배달원에게 주문 배정 |
| **검증항목** | 주문 상태 변경 |

#### Step 4: 다중 주문 배정
| 항목 | 내용 |
|------|------|
| **동작** | 체크박스로 여러 주문 선택 후 "선택 배정" |
| **API** | `POST /api/v1/drivers/assign-batch` |
| **기대결과** | 여러 주문 동시 배정 |
| **검증항목** | 배정 수 확인 |

#### Step 5: 배정 확인
| 항목 | 내용 |
|------|------|
| **동작** | "배정된 주문" 섹션 확인 |
| **API** | `GET /api/v1/drivers/my-orders` |
| **기대결과** | 배정된 주문 목록 표시 |
| **확인항목** | 상태 (order_confirmed) |

## 4. 시나리오 3: 식당 픽업

### 4.1 시나리오 설명
식당에서 음식을 수령하는 과정

### 4.2 테스트 단계

#### Step 1: 식당 이동
| 항목 | 내용 |
|------|------|
| **동작** | 배달 주소 확인 후 식당으로 이동 |
| **확인항목** | 식당 주소, 연락처 |
| **활용** | 네이버/카카오 지도 연동 |

#### Step 2: 식당 도착 확인
| 항목 | 내용 |
|------|------|
| **동작** | 식당 도착 후 조리 상태 확인 |
| **확인항목** | 예상 완료 시간 |
| **조치** | 지연 시 고객에게 안내 |

#### Step 3: 음식 수령
| 항목 | 내용 |
|------|------|
| **동작** | 조리 완료 후 음식 수령 |
| **확인항목** | 주문 내역과 일치 여부 |
| **확인항목** | 포장 상태 |

#### Step 4: 픽업 완료 처리
| 항목 | 내용 |
|------|------|
| **동작** | "배달 시작" 버튼 클릭 |
| **API** | `PUT /api/v1/orders/:id/delivering` |
| **기대결과** | 상태 `delivering`으로 변경 |
| **검증항목** | 픽업 시간 기록 |

## 5. 시나리오 4: 배달 수행

### 5.1 시나리오 설명
고객에게 음식을 배달하는 과정

### 5.2 테스트 단계

#### Step 1: 배달 주소 확인
| 항목 | 내용 |
|------|------|
| **동작** | 배달 주소 및 요청사항 확인 |
| **확인항목** | 상세 주소, 연락처, 출입 방법 |
| **활용** | 네비게이션 실행 |

#### Step 2: 고객 연락
| 항목 | 내용 |
|------|------|
| **동작** | 배달 전 고객에게 전화/SMS |
| **목적** | 부재 시 대처 방법 확인 |
| **확인항목** | 요청사항 재확인 |

#### Step 3: 배달 수행
| 항목 | 내용 |
|------|------|
| **동작** | 고객에게 음식 전달 |
| **확인항목** | 주문 내역 일치 여부 |
| **확인항목** | 결제 확인 (선결제인 경우) |

#### Step 4: 배달 완료 처리
| 항목 | 내용 |
|------|------|

| **API** | `PUT /api/v1/drivers/complete/:orderId` |
| **기대결과** | 상태 `completed`로 변경 |
| **검증항목** | 완료 시간 기록 |

#### Step 5: 완료 확인
| 항목 | 내용 |
|------|------|
| **동작** | 배달 내역에서 완료 확인 |
| **확인항목** | 배달비 누적 |
| **확인항목** | 총 배달 횟수 증가 |

## 6. 시나리오 5: 여러 주문 동시 처리

### 6.1 시나리오 설명
같은 방향의 여러 주문을 효율적으로 배달하는 경우

### 6.2 테스트 단계

#### Step 1: 다중 주문 배정
| 항목 | 내용 |
|------|------|
| **동작** | 인근 주소 주문 2~3건 선택 |
| **기대결과** | 일괄 배정 |

#### Step 2: 경로 최적화
| 항목 | 내용 |
|------|------|
| **동작** | 식당 위치와 배달 주소 기준 경로 설정 |
| **확인항목** | 식당 → 고객1 → 고객2 순서 |
| **활용** | 네비게이션 경로 설정 |

#### Step 3: 순차 픽업
| 항목 | 내용 |
|------|------|
| **동작** | 식당1에서 주문1 픽업, 식당2에서 주문2 픽업 |
| **확인항목** | 각 주문별 음식 확인 |

#### Step 4: 순차 배달
| 항목 | 내용 |
|------|------|
| **동작** | 고객1에게 배달 후 고객2에게 배달 |
| **API** | 각 주문별 완료 처리 |
| **기대결과** | 모든 주문 완료 |

## 7. 시나리오 6: 문제 상황 처리

### 7.1 시나리오 설명
배달 중 발생할 수 있는 문제 상황 대응

### 7.2 테스트 단계

#### Case 1: 고객 부재
| 항목 | 내용 |
|------|------|
| **상황** | 배달지에 고객 없음 |
| **동작** | 고객에게 전화 연락 |
| **조치** | 3회 이상 미응답 시 어드민 보고 |
| **기대결과** | 대기 안내 또는 재배달 |

#### Case 2: 식당 지연
| 항목 | 내용 |
|------|------|
| **상황** | 예상보다 조리 시간 초과 |
| **동작** | 고객에게 지연 안내 |
| **API** | SMS 알림 발송 |
| **기대결과** | 고객 이해, 대기 |

#### Case 3: 주소 불일치
| 항목 | 내용 |
|------|------|
| **상황** | 입력 주소와 실제 위치 상이 |
| **동작** | 고객에게 전화 확인 |
| **조치** | 주소 수정 요청 (어드민 통해) |

#### Case 4: 음식 문제
| 항목 | 내용 |
|------|------|
| **상황** | 배달 중 음식 훼손 |
| **동작** | 식당/어드민에게 보고 |
| **조치** | 재조리 또는 환불 |

#### Case 5: 교통 문제
| 항목 | 내용 |
|------|------|
| **상황** | 예상보다 배달 시간 초과 |
| **동작** | 고객에게 지연 안내 |
| **기대결과** | 고객 양해 |
